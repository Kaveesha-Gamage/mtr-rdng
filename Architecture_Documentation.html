Bulk Meter Reading Mobile Application

Target Framework: React Native (Expo SDK 54)  •  Status: Active Development

This document outlines the architecture, layout, data flows, and design patterns of the Bulk Meter Reading Mobile Application. The app is engineered to support utility workers performing bulk electricity/water meter readings, with a core focus on offline-first functionality, type-safety, and secure session handling.

1. Core Technology Stack

The application is built on top of a modern, reactive, and native-optimized stack:

  • Framework: React Native (v0.81) via Expo SDK 54 with the New Architecture enabled.
  • Language: TypeScript (v5.9) for build-time type verification.
  • Routing & Navigation: Expo Router (v6.0), using file-based routing and folder-grouped structures.
  • Local Offline Database: SQLite (via expo-sqlite), supporting relational tables for readings, bills, and customers.
  • Secure Cache & Sessions: Expo SecureStore (expo-secure-store) for encrypted, secure device storage of tokens and login user metadata.
  • Network Client: Axios client configured with timeout and custom headers for backend communication.

3. Architectural Layers

The app adheres to a clean, multi-layered architecture to separate concerns, improve testability, and simplify offline synchronization:

System Architecture Layers
  • Presentation Layer: Folder groups (auth) and (protected) segregate guest screens from protected zones. Authentication routing filters access reactive to Context.
  • State & Context Layer: Managed in AuthContext.tsx, allowing reactive, app-wide access to session metadata. Performs non-blocking SecureStore credentials check on cold starts.
  • Orchestration Layer: authService.ts validates inputs, formats requests, and manages the secure credentials lifecycle. pendingService.ts handles query coordinates, checking active reader tokens automatically.
  • Data Layer: Network APIs are structured under src/api returning strongly-typed JSON data. SQLite storage caches network inputs onto physical storage via the repository design pattern, preventing logical pollution in UI code.

5. Local SQLite Database Schema

The SQLite local database acts as the single source of truth for all query listings when the application operates offline.

Column Name SQLite Type Constraint Description
installationId TEXT PRIMARY KEY Unique hardware installation ID
accountNumber TEXT NOT NULL Utility customer account identifier
tariff TEXT - Customer tariff billing class
readerCode TEXT - Identifier mapping of the reader
dailyPack TEXT - Identifies sub-route assigned bundles
walkOrder TEXT - Visualized walking sequence for readers
currentBillCycle TEXT - Associated current billing cycle
areaCode TEXT - Assigned geographic operations area
customerCategory TEXT - Billing category (Domestic, Ind, etc)
previousReadingDate TEXT - Date index of previous billing reading
vatApplicable TEXT - Identifies taxation class mappings
currentReading INTEGER NULL Input reading value captured locally
remarks TEXT NULL Capture exceptions or remarks field
syncStatus TEXT DEFAULT 'PENDING' Upload tracker: 'PENDING' or 'SYNCED'
